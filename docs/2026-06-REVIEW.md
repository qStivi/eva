# Eva — Mid-2026 Reality Check & Direction

> Purpose: sanity-check the original Eva ideas (designed 2024–2025) against where AI

> machine first, server later.**

---

## 0. Developer intent (June 2026) — the frame for this review

1. **Desktop-first on this machine now.** It can run decent local models. Server/Proxmox
   deployment is still the long-term plan, but the immediate goal is a *working thing here*.
2. **Want Claude-style agent + tool use + parallelization.** Eva should *do things*, not just chat.
3. **Memory = relevance + persistence over months, but effectively infinite.** This is the
   reason Postgres + Chroma were put in place. It's a feature, not an accident.
4. **Local speech-to-speech not yet tested** on this hardware.

Target machine (from system notes): **AMD RX 9070 (RDNA4), 16 GB VRAM**, Bazzite (atomic Fedora/KDE),
**LM Studio already installed** (flatpak `ai.lmstudio.lm-studio`, which is llama.cpp + Vulkan).

---

## 1. Verdict in one paragraph

The **vision aged very well** and is *more* achievable today than when it was written. Several
ideas that the original docs treated as novel inventions ("two-track memory," "skill files,"
"goal/skill/desire," "held thoughts") have since become **named, productized, industry-standard
patterns** — that validates the thinking, but it means a few of them are now things to **reuse,
not re-invent**. A handful of **technical assumptions are dated** (Phi-3-mini, embedded
`transformers`, monthly-LoRA personalization, Coqui TTS, the token-savings rationale,
bespoke webhook/inter-assistant plumbing). The biggest *new* element vs. the original design is
the desire for **agentic behavior**, which forces one real decision: **pure-local vs. hybrid.**

## 2. This machine as the target (RDNA4 / 16 GB)

**Engine:** Use **llama.cpp with the Vulkan backend** (which is what LM Studio uses). On RDNA4,
Vulkan currently *beats* ROCm because RDNA4 (gfx1201) kernels aren't mature in the ROCm/vLLM
stack yet — measured ~62 t/s (llama-server Vulkan) vs ~48 t/s (vLLM ROCm) on a 9070 XT.
Revisit ROCm/vLLM in ~6–12 months; don't invest in it now.

**Models (16 GB sweet spot, replacing Phi-3-mini 3.8B):**
- **Qwen3** family — strongest for code + multilingual + tool-calling. Good default for an *agentic* Eva.
- **Llama 3.3 8B** — best all-round generalist.
- **Mistral Small 3** — throughput.
- **Phi-4** — strong small reasoning.
- Quantized ~30B is reachable with care, but leaves little headroom for voice/embeddings.

**Architectural consequence (important): desktop-first costs nothing long-term.**
Keep the FastAPI server. Run it on localhost. Make the LLM layer talk to an **OpenAI-compatible
endpoint** (LM Studio / `llama-server` / Ollama). The "desktop app" = Flutter **Linux desktop**
(or, honestly, keep the terminal UI for now) → localhost server. Moving to Proxmox later is a
**one-line base-URL change**. This also lets you **delete the embedded `transformers`/`torch`
path and the `USE_API` local/cloud fork** — there's now a single code path: "an OpenAI-compatible
URL," whether that points at LM Studio locally or a frontier API.

## 3. The four intents vs. 2026 reality

### 3.1 Desktop-first → ✅ correct, and well-supported
Covered in §2. No regret risk; it's the same server, just local. **Do this.**

### 3.2 "Claude-level agent + tool + parallelization" → the one real decision
**2026 reality:**
- Patterns are mature and documented: (a) parallel tool calls must be **both enabled and
  encouraged** — models default to sequential even when calls are independent; (b) an
  **orchestrator + specialized subagents** (search agent / generation agent / planner) can
  dramatically beat a single agent; (c) **only expose the tools each (sub)agent needs.**
- **MCP (Model Context Protocol)** is the standard tool/memory/integration layer. You already
  run an MCP server (SearXNG for LM Studio), so this is familiar ground.
- The **Claude Agent SDK** is purpose-built for orchestrator/subagent/parallel-tool patterns.

**The honest caveat / the decision:** agentic quality is **strongly model-dependent.** Claude is
exceptional at it; a local 8–14B on 16 GB is *usable* but not close. You cannot get
*Claude-level* agentic behavior **and** *fully-local-on-this-box* from one 16 GB model today.
Pick one consciously:

| Option | What it means | Trade |
|---|---|---|
| **A. Pure-local** | One local model does chat + tools | Private, free, offline; weaker multi-step tool use |
| **B. Hybrid (recommended for now)** | Local model for chat/memory; route hard agentic tasks to a frontier API (e.g. Claude) | Best capability; some calls leave the box; cost |
| **C. Local-now, API-optional later** | Build the tool/agent loop model-agnostically against an OpenAI-compatible API; run local by default, flip to frontier per-task | Keeps the door open; most flexible |

**Recommendation: design for C, run as A by default.** Build Eva's tool-loop and (eventually)
orchestrator against the OpenAI-compatible/MCP interface so the *brain* is swappable. That way
"local-only" and "borrow Claude for the hard stuff" are a config flag, not a rewrite — and it
matches the eventual server vision.

**Design impact:** Eva needs to evolve from a *chat loop with memory injection* into an
*agent loop* (model → tool calls → observations → repeat), with MCP as the tool surface. Your
journal, memory, and Home Assistant integrations become **MCP servers** that Eva (or any client,
incl. Claude Desktop / LM Studio) can consume — this replaces the bespoke "webhook bus" and
"API for other assistants" plumbing in the old design.

### 3.3 "Infinite" memory → your DB instinct was right; add the missing layer
**2026 reality:** This is now a named architecture. **Letta** (ex-MemGPT) implements your
two-track idea and extends it to effectively-infinite:
- **Memory blocks** = labeled, size-bounded sections of context (your Track 2, formalized).
- **Tiered storage** = in-context working memory ↔ external **recall** (recent history + vector
  search) ↔ **archival** (long-term facts, semantic retrieval). Your **Postgres + Chroma map
  directly onto recall + archival.**
- **Sleep-time compute** = a background agent that, while idle, **consolidates, summarizes, and
  rewrites** memory. *This is the piece the original Eva design is missing.*

**Key nuance:** "infinite" must mean **tiered + consolidated**, not "append everything forever and
stuff it into context." Raw infinite storage is trivial (a DB); infinite *useful* memory requires
**consolidation and forgetting**, or retrieval quality decays and cost grows unbounded. Your DB is
the right foundation; the upgrade is the **sleep-time consolidation job**.

**Two paths:**
- **Adopt Letta** (self-hostable, OSS, model-agnostic / OpenAI-compatible backend): get tiered
  memory + sleep-time + self-editing memory for free; cost is restructuring Eva around it.
- **Keep your stack**, add (a) a tiered retrieval policy and (b) a scheduled background
  "consolidator" that summarizes old conversations into durable facts and prunes/merges
  duplicates. More code, full control, keeps your two-track design intact.

Others worth a look: **Mem0** (personalization-focused, large community), **Zep** (temporal
knowledge graph — good if "what was true *when*" matters for a journal). **Recommendation:**
prototype against **Letta** at least once to feel the sleep-time pattern, then decide whether to
adopt it or port the idea into your own stack.

**Also consider:** with a single-user self-hosted app, **pgvector** can fold the vector store into
Postgres and drop Chroma as a separate moving part. Optional simplification, not urgent.

### 3.4 Local speech-to-speech → defer, and a pipeline likely fits better than pure S2S
**2026 reality:**
- **Moshi** (Kyutai): open, **full-duplex**, **sub-200 ms**, single-GPU/on-device. But:
  **English-only**, **~7B** (competes with your chat LLM for the same 16 GB), and it's an
  **end-to-end speech model** — it **bypasses your text pipeline**, so your two-track memory and
  character injection don't cleanly apply. Great demo, awkward fit for *this* architecture.
- **Pipeline path that preserves your design:** `faster-whisper` (STT) → your LLM → **Kokoro**
  (82M TTS, tiny VRAM) or **Kyutai Pocket TTS** (100M, realtime on **CPU**). ~500–800 ms,
  leaves the GPU for the model, and every word still flows through your memory/character logic.
- **Coqui TTS (in the old docs) is effectively dead** (company shut down) — replace with the above.

**Recommendation:** keep deferring voice. When you start, do the **pipeline**, not Moshi —
unless you specifically want the full-duplex "talk over each other" effect and are willing to
sacrifice memory/character integration and VRAM headroom.

## 4. Idea-by-idea scorecard (original design → 2026)

| Original idea | 2026 status | Action |
|---|---|---|
| Self-hosted, private, character-first companion | **Aged well**, still under-served niche | **Keep** — it's the real differentiator |
| Thin client / smart server (Flutter) | Still sound | **Keep** |
| Two-track memory as "the innovation" | **Validated but commoditized** (Letta/Mem0/Zep) | Keep concept; consider adopting a framework |
| "40–60% token savings" rationale | **Dated** (huge context windows + prompt caching) | Drop as the headline; reframe around relevance/persistence |
| "Skill files" system | **Prescient** — became Anthropic Agent Skills | Keep; align with the now-standard pattern |
| Goal/Skill/Desire system | Maps to modern agent state (Letta) | Keep the *social/internal-state* parts that are still novel |
| Held thoughts / inner monologue | Partly subsumed by reasoning models; **timing/hold-for-later still novel** | Keep the timing idea; lean on reasoning models for the rest |
| Adaptive learning = monthly LoRA | **Superseded** by memory + context engineering | De-prioritize fine-tuning; personalize via memory |
| Phi-3-mini 3.8B local | **Outclassed** | Move to Qwen3 / Llama 3.3 / Phi-4 |
| Embedded `transformers`/`torch` + `USE_API` fork | **Dated** | Single OpenAI-compatible client (local or API) |
| Webhook bus / inter-assistant API / platform adapters | **Largely subsumed by MCP** | Re-platform on MCP servers |
| Coqui TTS | **Dead** | Kokoro / Pocket TTS / faster-whisper pipeline |
| Chroma + Postgres + Redis (3 stores) | Fine; simplifiable | Optional: pgvector to fold vectors into Postgres |

## 5. Suggested near-term plan ("a working thing on this machine")

Ordered for fastest path to a daily-usable, *local* Eva:

1. **Engine swap (small, high-value):** point the LLM layer at an **OpenAI-compatible local
   endpoint** (LM Studio or `llama-server` with Vulkan). Delete the `transformers` path and
   the `USE_API` fork. Pick **Qwen3** (agentic) or **Llama 3.3 8B** (generalist) as default.
2. **Confirm the loop still works** end-to-end (terminal + WebSocket) on the local model.
3. **Make Eva agentic (MVP):** add a **tool-calling loop** and expose 1–2 tools via **MCP**
   (start with memory search + journal write). Enable *and encourage* parallel tool calls.
4. **Memory upgrade:** add a **background consolidator** (sleep-time-style) — summarize old
   conversations into durable facts, merge/prune duplicates — so "infinite" stays *useful*.
   (Or spike Letta to compare.)
5. **Desktop UI:** Flutter **Linux desktop** against the localhost WebSocket (or keep the
   terminal a bit longer — it already works; UI is not the bottleneck for "useful").
6. **(Later) Voice:** faster-whisper → LLM → Kokoro pipeline, only once 1–5 feel good.
7. **(Later) Server:** move the same server to Proxmox; change the base URL. Done.

**Decisions to make before/while doing the above:**
- **A vs B vs C** in §3.2 (local-purity vs. agentic capability). Recommended: **C, default A.**
- **Adopt Letta vs. extend your own memory** (§3.3). Recommended: **spike Letta, then decide.**
