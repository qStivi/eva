## Model evaluation (full 2026-06-13 gauntlet)
Each tested as Eva's brain via Letta — same quality conversation + memory checks + speed benchmark.
**Usable VRAM ≈ 14.9 GB** (desktop uses ~1 GB), so any 30B *always* needs CPU offload.

| Model | Decode | Saves on its own | Faithful | Perspective | Confab (chat) | Voice |
|---|---|---|---|---|---|---|
| **qwen3-8b** (fits VRAM) | **~36 t/s** | ✅ | ✅ | ❌ 1st-person | low (literal) | plain, loops |
| 30B-A3B-Instruct **@q3_k_l** | 36 t/s | ❌ **never fired** | — | — | ✅ worst | excellent |
| 30B-A3B-Instruct **@q4_k_m** | 19 t/s | ✅ | ✅ | ✅ 3rd-person | ⚠️ banter only (not saved) | **richest** |
| 30B-A3B-Thinking @q4_k_m | ~37 t/s* | ✅ | ✅ | ✅ | low | rich (+~7s think pause) |
| Qwen3.5-9B **@q8_0** | 25 t/s + thinking | ✅ | ✅ | ❌ 1st-person | low | ok, muddled |

\* same decode rate, but ~240 reasoning tokens/reply → ~8 s felt latency before the answer.

**Findings:**
- **Q3 quant breaks tool discipline** — 30B-Q3-Instruct *never* fired `memory_insert` AND confabulated a fake
  history. Q4 fixed it (real saves, 3rd-person, faithful). The quant matters for memory mechanics.
- **Confabulation in *banter* is partly inherent** to eloquent models (Q4-30B invents shared history as flavor)
  — but it does **not** poison the saved journal. Needs a guardrail (see NEXT).
- **3rd-person perspective: only the 30B got it right.** 8B and Qwen3.5-9B both write 1st-person → tracks with
  capability, not recency or thinking (the 8B already thinks and still writes 1st-person).
- **Qwen3.5-9B underwhelmed** — slower than the 8B (Q8 is bandwidth-heavy + new Gated-DeltaNet kernels
  unoptimised), thinking-on by default (`/no_think` doesn't disable it), 1st-person, muddled prose. Loads/runs
  fine though. Shelve until llama.cpp optimises; a fair retest = Q4_K_M + thinking-off.
- **Speed mechanics:** prefill ~280 t/s (uncached); LM Studio **caches the prompt prefix across turns**, so
  ongoing chat only re-processes new tokens (fast) — cold start / changed context pays the prefill.

**Decision (daily brain is a genuine speed-vs-quality trade):**
- **Daily driver = `qwen3-8b`** (fast, fits VRAM, reliable saves; accept 1st-person + loops for now). ← shipped.
- **Quality option = 30B-A3B-Instruct @q4_k_m** (richest voice + correct 3rd-person memory; 19 t/s; needs the
  confab guardrail). Revisit as daily driver after the **X3D CPU upgrade** speeds the offload path.
- **Sleep-time consolidator = 30B-A3B-Thinking @q4_k_m** (faithful + reasoning; latency irrelevant in background).
- **Avoid:** Q3 (broken memory); Q6/Q8 (slowest, no real quality gain on 16 GB); Qwen3.5-Q8 (slow + thinky).

**Models on disk** (`~/.var/app/ai.lmstudio.lm-studio/.lmstudio/models/lmstudio-community/`): qwen3-8b, nomic;
30B-A3B **Instruct & Thinking** each in **Q3_K_L / Q4_K_M / Q6_K**; **Qwen3.5-9B Q8_0**.

**Model ops gotchas:** download via **hf-mirror.com** + `aria2c -x4` (HF throttles — see [[hf-mirror...]] memory).
Download into `~/model-staging` then `mv` into the models dir — a *growing* file inside LM Studio's watched
folder makes its UI lag. Address models by **explicit `@quant`** (bare id is ambiguous → picks a half-downloaded
quant → crashes llama-server). I can drive `lms load <model> --gpu <ratio> -c <ctx> --parallel <n> --ttl <s>`
+ `lms unload --all` via the flatpak; advanced knobs (MoE-CPU offload count, KV-cache quant, Flash Attn) come
from the model's **saved GUI config**, not the CLI.

### Memory-fix recap

Timestamps (matters for "wasn't that last week?"): **messages** have a `date`; **core-memory
blocks/facts have NO timestamp**; **archival** passages are timestamped by design → store
episodic/dated memory in archival, stable facts in core (maps onto the future Logseq journal layer).

## Tools & guardrails roadmap (after memory works)
Risk ladder — build in this order, least privilege:
1. **Web search** — reuse the existing **SearXNG MCP** (`:3010`), point Letta at it. Safe, easy, first.
2. **Scoped journal/Logseq tools** — read/append/search the journal dir only (NOT generic file edit). The real vision.
3. **Shell** — only behind: sandbox (Letta container/distrobox, read-only mounts) + deny-rules + **HITL approval**
   (Letta natively supports "tool requires approval" — agent pauses; our client must handle the approval msg type).

**Classifier idea (like auto-mode):** Llama Guard 3/4 is self-hostable, tuned for tool calls, screens in/out,
includes Prompt Guard (injection detection). Use the small **Guard-3-1B** so it doesn't fight VRAM. Its
highest-value job is **prompt-injection screening once Eva ingests web/journal content** — not vetting our own
commands. It's one layer, not the whole defense.

## Decisions captured from this planning session

### Using the 46 GB system RAM (vs 16 GB VRAM)
Two genuinely useful uses:
1. **Run a model bigger than VRAM via CPU offload — especially a MoE.** e.g. **Qwen3-30B-A3B** (MoE, only
   ~3B active params) at Q4 (~18–20 GB): offload spillover layers to RAM; stays fast *because* MoE activates
   few params. Net: a noticeably smarter Eva than dense-14B, at acceptable speed. Worth A/B-ing against 14B.
2. **Pin auxiliary models to CPU/RAM** so the GPU is 100% the chat model: run the **nomic embedder** and a
   **Llama-Guard-1B** with GPU-offload=0 (CPU). 46 GB easily holds several resident models simultaneously
   (chat on GPU + embeddings + guard + a future sleep-time consolidation model on CPU) with no VRAM contention.

### Making Eva's file edits undoable (for the future file/journal tool)
**Make her writable area a git repo and have the file tool auto-commit each change.** Then every edit is a
revertible commit with a diff + history — natural fit for a journal, and matches your "git keeps the receipts"
ethos. Layer with the **HITL approval** above (show the diff before it commits) for review. Optionally have her
commit to a side branch you merge. (Logseq graphs are commonly git-versioned already.) Coarser fallback for
broader/system changes: the system disk is **btrfs** → snapshots, but git is the right tool for journal/notes.
